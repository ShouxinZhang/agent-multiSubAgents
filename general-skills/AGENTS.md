# 仓库 Agent 指令（通用版）

## 语言与沟通

- 默认跟随用户语言。
- 解释决策时优先说明业务价值、收益、风险和影响范围。

## 核心工程原则

- 优先级：业务结果 > 长期架构 > 表达简洁。
- 先读现有代码再改动，避免重复建设。
- 保持最小改动，禁止未经授权扩 scope。
- 维护模块边界，避免跨模块污染。
- 新增业务代码优先落在叶子目录。

## AGENTS 与 Skills 的配对关系

`AGENTS.md` 是执行契约，Skill 是能力载体。

配对规则：

1. 仅使用一个顶层入口 Skill：`local-dev-workflow`。
2. 其他能力通过 `local-dev-workflow/subskills/` 按条件触发。
3. 同一能力的执行规则只在对应 subskill 中维护，避免多文档重复。
4. 若用户显式指定某能力，仍通过入口 Skill 路由到对应 subskill 执行。

## 强制工作流（单入口）

每个开发任务都先走 `local-dev-workflow`，并按条件触发以下 subskills：

- `build-check`：代码改动后执行质量门禁。
- `dev-logs`：每个开发循环结束记录日志。
- `repo-structure-sync`：结构变更后同步仓库结构文档。
- `git-management`：阶段成果或大改前后建立可回滚锚点。
- `domain-data-update`：修改核心领域数据时执行一致性与回滚约束。
- `modularization-governance`：边界治理、重构验收、架构风险控制。
- `dependency-review-system`：高风险变更评审与统一裁决。

## 人类意图对齐

任何代码变更前，先确认：

- 将修改/新增的文件
- 核心实现方式
- 影响范围与主要风险

确认清晰后再实施。

## 变更前检查

- 每次变更前先读取 `AGENTS.md`。
- 优先使用 MCP 获取模块上下文，架构文档作为补充。
- 涉及删除/回滚操作前先创建备份。

## 质量与验证

- 禁止无验证交付。
- 代码变更后执行：

```bash
bash scripts/check_errors.sh
```

- 需要时执行：

```bash
npm test
```

## 文档与可追溯性

- 结构变更后保持架构文档与仓库一致。
- 每个开发循环写入 `docs/dev_logs/{YYYY-MM-DD}/`。
- 日志必须包含：用户请求、秒级时间戳、文件清单、变更说明、验证结果。

## 自动化与版本策略

- 可自动化步骤必须脚本化。
- SDK/依赖优先采用可验证的稳定版本。

## 禁止行为

- 未跑检查/测试就提交。
- 未更新 dev log 就结束开发。
- 结构变更后不更新结构文档。
- 让用户手工执行可脚本化工作。
